Prism Behavior Trees Documentation

This document provides a comprehensive guide to using Behavior Trees in the Prism framework. Behavior Trees are a powerful way to model AI decision-making in games, allowing for modular, reusable, and maintainable AI logic.

Overview

Behavior Trees in Prism are composed of nodes that represent tasks, decisions, or composites. Each node returns a status: true (success), false (failure), or an Action object (running task). The tree ticks every frame, evaluating nodes until an Action is returned or the tree completes.

Key concepts:

  • Leaf Nodes: Perform actions or checks (e.g., move, attack, check conditions).
  • Composite Nodes: Control execution flow (e.g., Sequence, Selector).
  • Decorators: Modify node behavior (e.g., Succeeder, Conditional).
  • Root Node: The entry point of the tree, must return an Action.

Core Components

BehaviorTree.Node

The base class for all nodes. Extend it to create custom behaviors.

local MyNode = prism.BehaviorTree.Node:extend("MyNode")

function MyNode:run(level, actor, controller)
    -- Return true (success), false (failure), or an Action
    return true
end

BehaviorTree.Root

The root of the tree. Runs children sequentially until one returns an Action, which it then returns. If no Action is found, it errors.

prism.BehaviorTree.Root({
    child1,
    child2,
    -- ...
})

BehaviorTree.Sequence

Runs children in order. Succeeds if all children succeed. Fails if any child fails. Returns an Action immediately if a child does.

Use for: Chained tasks that must all complete (e.g., find target, then move, then attack).

prism.BehaviorTree.Sequence({
    prism.behaviours.FindTargetBehaviour(),
    prism.behaviours.MoveBehaviour(),
    prism.behaviours.AttackBehaviour(),
})

BehaviorTree.Selector

Runs children in order. Succeeds and returns the result of the first child that succeeds. Fails if all children fail.

Use for: Fallback options (e.g., try to eat, if no food, hunt).

prism.BehaviorTree.Selector({
    prism.behaviours.EatBehaviour(),
    prism.behaviours.HuntBehaviour(),
    prism.behaviours.WaitBehaviour(),
})

BehaviorTree.Succeeder

Always succeeds, regardless of the child's result. Useful for ignoring failures.

prism.BehaviorTree.Succeeder(prism.behaviours.RiskyBehaviour())

BehaviorTree.Conditional

Runs a condition function. Succeeds if the function returns true, fails otherwise.

prism.BehaviorTree.Conditional(function(level, actor, controller)
    return actor:get(prism.components.Health).current > 50
end)

How Behavior Trees Work

Behavior Trees are evaluated every frame via the BTController component. The tree starts at the Root, which ticks its children. Execution flows based on node types:

  • Ticking: Nodes are reevaluated each frame, allowing dynamic behavior.
  • Blackboard: Use controller.blackboard (short-term) and controller.blackboard.long (long-term) to store state between ticks.
  • Actions: When a node returns an Action, the tree stops and the Action is executed. Actions represent ongoing tasks.

Example flow:

In a Sequence: Check health → If low, find cover → Move to cover. If any step fails, the sequence fails.

In a Selector: Try to attack → If no target, flee → If can't flee, wait.

Examples

Simple Kobold AI

A kobold that prioritizes hunger, then hunting, otherwise waits.

prism.components.BTController(prism.BehaviorTree.Root({
    prism.BehaviorTree.Selector({
        -- Hunger Subroutine
        prism.BehaviorTree.Sequence({
            prism.behaviours.SatietyBelowPercentageCheckBehaviour(90),
            prism.behaviours.FindEdibleBehaviour(),
            prism.BehaviorTree.Selector({
                prism.BehaviorTree.Sequence({
                    prism.behaviours.CheckTargetInRangeBehaviour(0),
                    prism.behaviours.EatBehaviour(),
                }),
                prism.behaviours.MoveBehaviour(0),
            }),
        }),
        -- Hunt Subroutine
        prism.BehaviorTree.Sequence({
            prism.behaviours.FindEnemyBehaviour(),
            prism.BehaviorTree.Selector({
                prism.BehaviorTree.Sequence({
                    prism.behaviours.CheckRoomTargetBehaviour(),
                    prism.behaviours.MoveBehaviour(),
                }),
                prism.BehaviorTree.Sequence({
                    prism.behaviours.HPBelowPercentageCheckBehaviour(34),
                    prism.behaviours.FindNearestRoomNotContainingEnemyBehaviour(),
                    prism.behaviours.MoveBehaviour(),
                }),
                prism.BehaviorTree.Sequence({
                    prism.behaviours.CheckTargetInRangeBehaviour(1),
                    prism.behaviours.AttackBehaviour(),
                }),
                prism.behaviours.MoveBehaviour(),
                prism.behaviours.WaitBehaviour(),
            }),
        }),
    }),
    prism.behaviours.WaitBehaviour(),
}))

Custom Behavior

Create a custom node for patrolling.

local PatrolBehaviour = prism.BehaviorTree.Node:extend("PatrolBehaviour")

function PatrolBehaviour:run(level, actor, controller)
    local path = controller.blackboard.short["patrol_path"]
    if not path then
        path = generate_random_path(actor.position)
        controller.blackboard.short["patrol_path"] = path
    end
    if at_end_of_path(actor.position, path) then
        return true  -- Patrol complete
    end
    return prism.actions.MoveAction(path[next_point])
end

Best Practices

  • Modularity: Break complex behaviors into small, reusable nodes. Avoid monolithic trees.
  • Prioritization: Use Selectors for priority (e.g., survival over aggression). Sequences for required steps.
  • State Management: Use the blackboard for communication between nodes. Prefer short-term for per-tick data.
  • Performance: Avoid expensive operations in frequently ticked nodes. Cache results where possible.
  • Debugging: Log node results during development. Ensure trees don't loop infinitely.
  • Readability: Use comments to describe subroutines. Consistent indentation and naming.
  • Testing: Unit test individual nodes. Simulate tree execution with mock actors/levels.
  • Avoid Deep Nesting: Flatten trees where possible. Use decorators to simplify.
  • Dynamic Behavior: Leverage senses and components for reactive AI (e.g., sight, hearing).
  • Balance: Ensure trees don't lead to exploitable or boring behavior. Tune percentages and ranges.

Code Style Guidelines

  • Use prism.behaviours for standard behaviors; custom nodes in modules/game/behaviours/.
  • Name nodes descriptively (e.g., FindEnemyBehaviour not FEB).
  • Return consistent types: boolean for checks, Action for tasks.
  • Avoid side effects in run methods; use blackboard or components.
  • Document parameters and return types with EmmyLua comments.
  • Keep nodes stateless; rely on actor components and blackboard.

For more examples, refer to existing actors in modules/game/actors/. Contributions to this documentation are welcome via pull requests.
